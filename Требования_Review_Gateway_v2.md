# Требования к сервису Review Gateway

Версия: 2.0 (скорректирована по итоговой архитектуре AI Code Review Platform)

## Назначение

Review Gateway — центральный сервис AI Code Review Platform.

Сервис принимает запросы на выполнение AI Code Review из GitLab CI, управляет жизненным циклом ревью, распределяет задачи между LLM-серверами через пул Worker, сохраняет результаты и публикует комментарии в Merge Request.

Review Gateway — единственная точка входа в систему и единственный владелец бизнес-логики и состояния.

Расчётный масштаб: 20–30 MR в день, длительные LLM-задачи (до десятков минут), 1–10 LLM backend-серверов.

---

# 1. Функциональные требования

## 1.1 Приём запросов

Сервис должен:

* принимать запрос на создание нового AI Review;
* принимать данные Merge Request, необходимые для выполнения анализа (project id, MR IID, head SHA, base SHA, diff, версия промпта);
* **валидировать размер diff относительно допустимого лимита контекста LLM и отклонять запрос с превышением, возвращая явный код ошибки (`DIFF_TOO_LARGE`), пригодный для публикации понятного сообщения в MR средствами CI**;
* возвращать уникальный идентификатор Review;
* завершать обработку запроса без ожидания окончания генерации ответа LLM.

## 1.2 Хранение входных данных

Сервис должен сохранять:

* информацию о Merge Request;
* head SHA и base SHA;
* diff;
* используемую версию промпта.

Входные данные неизменяемы после создания. Сохранённые входные данные должны позволять повторно выполнить Review без обращения к GitLab.

## 1.3 Управление очередью

Сервис должен:

* помещать новые Review в очередь;
* выбирать задачи согласно приоритету и времени создания;
* гарантировать выдачу каждой задачи ровно одному Worker;
* автоматически возвращать задачу в очередь при сбое Worker.

## 1.4 Управление жизненным циклом Review

Сервис должен поддерживать состояния Review:

* NEW
* QUEUED
* RUNNING
* COMPLETED
* PUBLISHED
* FAILED
* CANCELLED
* OBSOLETE

Все переходы между состояниями выполняются только сервисом Review Gateway.

**Сервис должен предоставлять операцию ручной отмены Review администратором. Отменённый Review переводится в состояние CANCELLED; выполняющаяся задача прекращается механизмом, описанным в п. 1.7.**

## 1.5 Дедупликация

Ключ дедупликации: project id + Merge Request IID + head SHA.

При поступлении запроса сервис должен:

* **не создавать новый Review, если по ключу существует Review в состоянии NEW, QUEUED, RUNNING, COMPLETED или PUBLISHED** (в том числе при повторном запуске pipeline на том же коммите);
* возвращать идентификатор существующего Review;
* создавать новый Review только если предыдущие по ключу находятся в состояниях FAILED, CANCELLED или OBSOLETE.

При поступлении нового head SHA для того же Merge Request сервис должен:

* перевести все незавершённые (не PUBLISHED) Review этого MR в состояние OBSOLETE;
* гарантировать, что результаты OBSOLETE Review не публикуются.

## 1.6 Управление Backend

Сервис должен:

* хранить реестр Backend (адрес, модель, ёмкость, статус);
* определять Backend для выполнения Review с учётом его состояния и свободной ёмкости;
* исключать из назначения Backend в состояниях SUSPECT, MAINTENANCE, OFFLINE;
* переводить Backend в SUSPECT при недоступности или сбое;
* **периодически проверять доступность Backend в состоянии SUSPECT и автоматически возвращать восстановленный Backend в состояние ACTIVE**.

Занятость Backend вычисляется по фактическому числу выполняющихся задач; отдельные счётчики занятости не ведутся.

## 1.7 Контроль выполнения

Сервис должен:

* отслеживать выполнение Review;
* принимать heartbeat от Worker с заданной периодичностью;
* **в ответе на heartbeat сообщать Worker о необходимости прекратить выполнение, если Review утратил актуальность (OBSOLETE) или отменён (CANCELLED)** — Worker обязан прервать генерацию и перейти к следующей задаче;
* обнаруживать зависшие задачи по отсутствию heartbeat в течение заданного интервала;
* дополнительно ограничивать максимальную общую длительность выполнения как страховочный механизм;
* возвращать зависшую задачу в очередь.

## 1.8 Повторное выполнение

Сервис должен:

* выполнять повторный запуск Review при временной ошибке (сбой Worker, недоступность Backend, отсутствие heartbeat);
* ограничивать количество попыток выполнения;
* переводить Review в FAILED после превышения лимита попыток.

Retry реализуется только на стороне Gateway; Worker и Backend не содержат собственной логики повторов.

## 1.9 Приём результатов

Сервис должен:

* принимать результат выполнения от Worker;
* **сохранять исходный (сырой) ответ модели до его разбора на комментарии**;
* разбирать ответ на структурированные комментарии (файл, строка, важность, текст);
* сохранять метрики выполнения (токены, длительность, модель);
* **принимать повторную доставку результата идемпотентно: если задача уже не находится в состоянии RUNNING, повторный результат подтверждается без изменения состояния системы**.

## 1.10 Публикация результатов

После успешного завершения Review сервис должен:

* опубликовать комментарии через GitLab API;
* фиксировать факт публикации для каждого комментария отдельно;
* при повторной публикации отправлять только неопубликованные комментарии;
* перед публикацией проверять актуальность Review (не OBSOLETE, не CANCELLED);
* при временной недоступности GitLab сохранять Review в состоянии COMPLETED и повторять публикацию позднее.

## 1.11 Сбор статистики

Сервис должен сохранять:

* время ожидания в очереди;
* время выполнения;
* количество токенов (prompt / completion);
* используемый Backend и модель;
* количество найденных замечаний;
* итоговый статус Review;
* все значимые события жизненного цикла (создание, взятие в работу, retry, завершение, публикация, устаревание) в журнале событий.

# 2. Технические требования

## 2.1 Технологический стек Gateway

* Java 21 LTS
* Spring Boot 3.x
* Flyway для миграций схемы БД

Обоснование выбора Java 21 при основном стеке команды Java 17: виртуальные потоки соответствуют профилю нагрузки (большое число длительных ожиданий HTTP), LTS-статус обеспечивает сопровождаемость. Решение фиксируется осознанно и согласуется с командой сопровождения.

## 2.2 Хранение данных

Единственное хранилище — PostgreSQL.

В PostgreSQL хранятся: очередь, входные данные, результаты (включая сырой ответ модели), комментарии, события, реестр Backend, статистика.

Использование Redis, Kafka и других систем хранения не предусматривается.

## 2.3 Очередь

Очередь реализуется средствами PostgreSQL.

Гарантия выдачи задачи ровно одному Worker обеспечивается на уровне СУБД. Длительные транзакции и удержание блокировок на время генерации не допускаются: захват задачи выполняется короткой транзакцией, владение задачей далее подтверждается состоянием RUNNING и heartbeat.

## 2.4 Формат API

Внешний API: REST, формат данных — JSON.

Основные операции:

* `POST /reviews` — создание Review (GitLab CI);
* `GET /reviews/{id}` — статус Review;
* `POST /jobs/claim` — получение задачи Worker;
* `POST /jobs/{id}/heartbeat` — подтверждение выполнения; ответ содержит признак продолжения;
* `POST /jobs/{id}/result` — передача результата;
* `DELETE /reviews/{id}` — ручная отмена (администратор);
* `GET /health`, `GET /metrics`, `GET /backends` — служебные.

## 2.5 Worker

Worker — отдельное stateless-приложение (Java 21, единый стек сопровождения).

Развёртывание: по одному Worker на каждый LLM-сервер (Mac mini), запуск через launchd, обращение к llama-server по localhost, к Gateway — по REST.

Worker взаимодействует с Gateway только через REST API и не имеет доступа к PostgreSQL.

Worker не должен обращаться к GitLab и не должен изменять бизнес-состояния Review.

Worker обязан прекращать генерацию при получении признака остановки в ответе на heartbeat.

## 2.6 Backend

Backend — HTTP API, совместимый с OpenAI Chat Completions.

На первом этапе поддерживается только llama-server. Backend не содержит никакой логики системы.

## 2.7 Отказоустойчивость

Gateway должен корректно восстанавливать работу после перезапуска.

После запуска сервис продолжает обработку незавершённых Review. **Задачи в состоянии RUNNING при рестарте Gateway не сбрасываются и не перезапускаются: Worker продолжает выполнение, контроль осуществляется штатным heartbeat-механизмом.**

Все фоновые процессы (контроль таймаутов, публикация, проверка Backend) должны быть идемпотентными.

## 2.8 Развёртывание

Gateway разворачивается в одном экземпляре как systemd-сервис с автоматическим перезапуском.

Резервное копирование PostgreSQL (`pg_dump`) выполняется по расписанию.

## 2.9 Логирование и события

Все операции журналируются с указанием Review ID, Worker, Backend, длительности и статуса.

Значимые изменения состояния дополнительно фиксируются в журнале событий в БД независимо от файловых логов.

# 3. Безопасность

* Запросы GitLab CI к Gateway аутентифицируются токеном (Bearer).
* Запросы Worker к Gateway аутентифицируются отдельным токеном; токены CI и Worker различаются и разграничивают доступные операции.
* GitLab API Token для публикации комментариев хранится только в Gateway.
* Worker не имеет доступа к GitLab-токенам и учётным данным PostgreSQL.
* Секреты не передаются во входных данных Review и не сохраняются в журнале событий.

# 4. Логические требования

## 4.1 Единственная точка входа

Все операции с Review выполняются только через Review Gateway. Прямое изменение состояния Review другими компонентами не допускается.

## 4.2 Единственный источник истины

Единственный источник состояния системы — PostgreSQL. Состояние не хранится в памяти приложения.

## 4.3 Неизменяемость входных данных

После создания Review входные данные не изменяются. Повторный запуск Review всегда использует сохранённый payload.

## 4.4 Идемпотентность

* Повторный запрос на создание Review по тому же ключу дедупликации не создаёт новую задачу.
* Повторная доставка результата от Worker не изменяет состояние завершённой задачи.
* Повторная публикация не создаёт дубликатов комментариев.

## 4.5 Разделение ответственности

| Компонент | Ответственность |
| --- | --- |
| GitLab CI | получить diff и отправить запрос |
| Review Gateway | вся бизнес-логика и состояние |
| PostgreSQL | хранение состояния и истории |
| Worker | выполнение LLM-задачи |
| llama-server | генерация текста |
| GitLab API | публикация комментариев в MR |

## 4.6 Независимость компонентов

Worker не знает: GitLab, очередь, состояния Review, retry, дедупликацию, схему БД.

Backend не знает: GitLab, Review, очередь, Worker.

## 4.7 Воспроизводимость

Любой Review может быть повторно выполнен и разобран на основании данных, сохранённых в PostgreSQL (входной payload + сырой ответ модели).

## 4.8 Простота масштабирования

Добавление нового Backend требует только регистрации в БД и запуска Worker на новом сервере. GitLab CI и API Gateway при этом не изменяются.

# Нефункциональные принципы разработки

* **Один сервис — одна ответственность.** Gateway управляет жизненным циклом ревью и не выполняет генерацию текста.
* **Одна бизнес-сущность — один источник истины.** Любое состояние Review хранится только в PostgreSQL.
* **Минимум инфраструктуры.** Redis, Kafka, брокеры сообщений, Prometheus, Kubernetes не используются до появления реальной потребности.
* **Явные состояния вместо скрытой логики.** Все изменения статуса выполняются через определённые переходы state machine; состояний без входных переходов не существует.
* **Простые синхронные интерфейсы.** Взаимодействие компонентов — обычный REST без дополнительных транспортов.
* **Идемпотентность всех повторяемых операций.** Создание, доставка результата, публикация, фоновые процессы.
* **Fail fast на входе.** Некорректные или невыполнимые запросы (превышение размера) отклоняются при приёме, а не в очереди.
* **Минимум магии.** Простые SQL-запросы и понятная бизнес-логика вместо сложных фреймворков и скрытых механизмов.
